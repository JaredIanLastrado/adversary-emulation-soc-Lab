# Slack Notification Design

## Purpose

Slack provides operational visibility into AESOC alerts, cases, and tickets.

Messages are generated by Shuffle and posted by the **AESOC SOAR Agent**.

Slack is not the official investigation or ticket record.

## Channel Design

| Channel | Purpose |
|---|---|
| `#aesoc-alerts` | Wazuh alert notifications |
| `#aesoc-cases` | TheHive case lifecycle and handback notifications |
| `#aesoc-tickets` | Zammad ticket creation and completion notifications |

## Notification Flow

```text
Wazuh, TheHive, or Zammad Event
        ↓
Shuffle
        ↓
Formatted Slack Message
        ↓
Appropriate AESOC Channel
```

## Alert Notifications

Alert messages may include:

- Severity
- Wazuh rule level
- Alert title
- Rule ID
- Affected endpoint
- MITRE ATT&CK mapping
- Recommended next step

## Case Notifications

Case messages may include:

- Case title
- Case number
- Severity
- Status
- Assigned analyst
- Required next action

## Ticket Notifications

Ticket messages may include:

- Zammad ticket number
- Ticket title
- Ticket group
- Ticket state
- Linked TheHive case
- Assigned owner
- Completion timestamp

## Design Principles

- Notify only meaningful lifecycle events.
- Keep detailed findings in TheHive.
- Keep assigned work details in Zammad.
- Avoid unnecessary in-progress notifications.
- Identify the next owner or expected action.
- Do not use Slack to approve containment or case closure.

## System Boundaries

| Information | Primary location |
|---|---|
| Alert and investigation details | TheHive |
| Assigned work | Zammad |
| Workflow execution | Shuffle |
| Operational visibility | Slack |

## Security Notes

- Slack tokens and webhook URLs are not published.
- Messages should not contain credentials or sensitive evidence.
- Public screenshots use controlled lab data.
